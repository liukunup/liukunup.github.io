低功耗模式与语音唤醒

约 2237 字大约 7 分钟

2026-06-16

引言

"Hey Siri" 响应的那一刻，手机已经一直在听。在电池供电的设备上，要实现 7×24 小时的持续监听，整个唤醒系统的平均功耗必须控制在 mW 甚至 µW 级。

这是一场"一直听"和"不掉电"之间的工程博弈。

1. 系统的功耗分布

以智能手机为例，各模块的典型功耗：

模块典型功耗备注
屏幕亮起500-1500 mW最大耗电来源
主 CPU 运行1000-3000 mWA核全速
主 CPU 待机50-200 mW浅睡眠
DSP / 协处理器1-10 mW低功耗音频处理
模拟麦克风 (PDM)0.5-2 mW始终开启
语音唤醒专用硬件0.1-1 mW定制 ASIC

语音唤醒的目标：在检测到唤醒词之前，系统总功耗 < 10 mW。

2. 经典架构对比

纯 CPU 方案（不可取）

麦克风 → Codec → CPU (持续运行) → 唤醒

功耗: > 100 mW
CPU 无法进入深度睡眠，电池很快耗尽。

DSP + CPU 两级方案（主流）

麦克风 → Codec → DSP (常开) → 检测到候选 → CPU (验证) → 唤醒

功耗: ~5 mW (DSP 常开) + ~200 mW (CPU 瞬启)

专用唤醒芯片（最优）

麦克风 → 专用 Wake-Up ASIC → 检测到唤醒词 → CPU → 唤醒

功耗: < 1 mW (常开)

3.2 Level 1 — DSP 持续监听

组件作用功耗
麦克风阵列采集音频1-2 mW
Codec (ADC)模拟→数字2-5 mW
音频缓冲区DMA 直接搬运，不经过 CPU< 0.1 mW
DSP 核轻量模型推理1-5 mW
SRAM模型参数 + 特征缓冲区< 0.5 mW

DSP 上运行的模型特征：

  • 参数量：10K-50K
  • 量化精度：INT8
  • 特征：MFCC（10-13 维）
  • 帧率：每 10-20ms 推理一次
  • 输出：唤醒词置信度 + 粗略语音活动检测

常见 DSP：

DSP 平台厂商典型功耗算力
Hexagon (CDSP)Qualcomm2-8 mW中等
Andes N22Andes Tech0.5-2 mW低
Cadence HiFi-3Cadence3-10 mW中高
Arm Cortex-M55Arm1-5 mW中等
Hifi MiniCadence0.3-1 mW低

3.3 Level 2 — CPU 验证

DSP 检测到候选唤醒事件后，通过硬件中断唤醒主 CPU：

DSP 触发中断
    ▼
CPU 从深度睡眠恢复 (10-50ms)
    ▼
加载大模型 (200K-500K params)
    ▼
读取 DSP 保留的音频缓冲区 (1-2s)
    ▼
执行精度验证推理 (FP32 / FP16)
    ▼
验证通过 → 系统唤醒
验证失败 → CPU 返回睡眠

为什么需要 Level 2 验证：

  • DSP 模型小、精度有限，可能有误触发
  • 唤醒 CPU 后需要更精确的二次确认
  • 降低误唤醒概率，提升用户体验

4. 低功耗设计关键技术

4.1 电源域隔离

将始终开启的唤醒模块放在独立的电源域中，与该电源域外的模块隔离：

                    ┌── VDD_WAKE (常开) ──┐
                    │  麦克风 + Codec      │
                    │  + DSP + SRAM       │ 0.8V
                    └─────────────────────┘
                    ┌── VDD_SYS (可关断) ──┐
                    │  CPU + GPU + NPU    │ 关断时漏电 0
                    └─────────────────────┘

4.2 时钟门控与频率缩放

  • DSP 运行在低频（如 32kHz - 50MHz）
  • 没有语音活动时，DSP 进入停顿状态（WFI），等待 VAD 触发
  • CPU 睡眠时关闭 PLL（锁相环），仅保留 32kHz RTC
// DSP 最低功耗循环伪代码
void always_on_loop() {
    while (1) {
        // 等待音频 DMA 完成
        __WFI();  // Wait For Interrupt，功耗降至 µA 级

        // 音频数据就绪
        process_audio_buffer();

        // VAD 检测
        if (!vad_detected()) {
            continue;  // 无语音，回到 WFI
        }

        // 提取 MFCC
        extract_features();

        // 模型推理
        float score = run_model();

        if (score > THRESHOLD_L1) {
            trigger_cpu_wakeup();
        }
    }
}

4.3 麦克风选择

类型功耗信噪比适用场景
ECM (驻极体)0.5-1 mA60-70 dB传统方案
MEMS (模拟)100-300 µA60-65 dB低成本
MEMS (数字 PDM)300-800 µA63-70 dB主流选择
PDM 多麦克风每路 ~200 µA65-70 dB波束成形阵列

MEMS PDM 麦克风是端侧语音唤醒的主流选择：

  • 超低功耗（~0.5 mW）
  • 数字输出，抗干扰好
  • 尺寸小（2.75 × 1.85 mm）

4.4 音频缓冲区优化

音频缓冲区通常由 DMA (Direct Memory Access) 直接管理，不需要 CPU 参与：

麦克风 PDM
    ↓
PDM → PCM 转换 (硬件)
    ↓
DMA 写入 环形缓冲区 (SRAM)
    ↓
DSP 读取最新帧

CPU / DSP 在数据就绪前保持睡眠

环形缓冲区：

#define BUFFER_SIZE   (16000 * 2)  // 2秒音频 @16kHz

int16_t audio_buffer[BUFFER_SIZE];
volatile uint32_t write_pos = 0;

// DMA 中断 —— 新数据到达
void dma_complete_isr() {
    int16_t samples[160];  // 10ms 数据
    dma_read(samples, 160);

    for (int i = 0; i < 160; i++) {
        audio_buffer[(write_pos + i) % BUFFER_SIZE] = samples[i];
    }
    write_pos = (write_pos + 160) % BUFFER_SIZE;

    // 唤醒 DSP（如果 DSP 在 WFI 状态）
    dsp_wakeup();
}

4.5 模型量化与剪枝

量化：

精度模型大小推理功耗精度损失
FP32400 KB100%基准
FP16200 KB~60%~0%
INT8100 KB~25%< 1%
INT450 KB~15%1-3%

剪枝：移除权重接近 0 的连接，可压缩 50-80%。

知识蒸馏：用大模型（Teacher）监督小模型（Student）训练，小模型获得接近大模型的精度。

6. 实测功耗分解

以典型智能手机方案为例的实测功耗：

条件: iPhone 待机 + 语音唤醒开启
------------------------------------------------
总系统功耗（唤醒模块）: 4.8 mW

其中:
  麦克风 (MEMS x2)       0.6 mW  (12.5%)
  音频 Codec             1.2 mW  (25%)
  DSP + SRAM            2.0 mW  (41.7%)
  SoC 接口 (I2S/DMA)    0.5 mW  (10.4%)
  其他 (LDO, 时钟等)     0.5 mW  (10.4%)
------------------------------------------------

唤醒瞬间功耗峰值：

DSP 检测到候选 → CPU 恢复供电
   CPU 启动 (PMIC ramp-up)   5-10ms   5-10 mJ
   CPU 加载模型              10-30ms  5-15 mJ
   CPU Level 2 推理          20-50ms  10-25 mJ
   确认唤醒 / 放弃           50-100ms 20-50 mJ
   CPU 返回睡眠              5-10ms   1-2 mJ

如为误触发 (Level 1 假阳性): 每次约 30-60 mJ
如 24h 误触发 10 次: 约 0.3-0.6 mAh（可忽略）

关键点：Level 1 模型宁可高召回（不漏唤醒词）也可以接受一定高误报，因为误报的代价（CPU 瞬启验证）远低于漏报的代价（用户觉得设备不灵敏）。

7. 优化策略总结

策略说明效果
两级唤醒DSP 粗筛 + CPU 精验常开功耗降低 10-50 倍
电源域隔离关断非必要模块静态漏电归零
时钟缩放最低工作频率动态功耗与频率成正比
WFI 停顿无数据时 DSP 暂停空闲功耗降至 µA 级
DMA 直传音频不经过 CPUCPU 保持睡眠
INT8 量化小模型低精度推理模型大小 ×0.25，速度 ×4
模型剪枝移除冗余权重推理量减少 50-80%
先 VAD 后 KWS无语音时不推理 KWS减少无效推理 60-80%
自适应阈值根据环境噪声调整降低噪音下误唤醒
麦克风数量1 颗 vs 多颗阵列每颗~0.3 mW

VAD 前置过滤效果

持续 KWS 推理:    100% 时间在推理 (10ms 每帧)
VAD + KWS 推理:   20-30% 时间在推理 (仅语音段推理)

功耗比例:         30% → 100%（取决于说话密度）
日常大部分时间静音，VAD 前置可节能 70-80%。

8. 未来趋势

趋势说明
存内计算 (Computing in Memory)在 SRAM 内直接完成矩阵乘法，消除冯诺依曼瓶颈
事件驱动传感器麦克风本身集成简单事件检测，无事件时不输出数据
亚阈值电路设计工作在晶体管亚阈值区，功耗极低但速度慢（够用即可）
自适应电压缩放 (AVS)根据温度/工艺角动态调压
片上学习DSP 上在线微调，适应使用者的声音和环境
多模态唤醒语音 + 加速度（敲击）+ 接近传感器联合低功耗唤醒

存内计算是语音唤醒功耗的下一个量级突破点。现有方案的 DSP + SRAM 消耗大量功耗在数据搬运上（数据从 SRAM 到 ALU），而存内计算直接在存储阵列中完成乘加运算，功耗可再降低 10 倍以上。
