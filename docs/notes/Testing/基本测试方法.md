## 测试金字塔

测试金字塔是指导测试分层的基本原则，从底向上逐层递增，推荐比例约为 **70:20:10**。

```
        /\
       /  \        E2E 测试（少量）
      /    \
     /──────\
    /        \     集成测试（适中）
   /          \
  /────────────\
 /              \  单元测试（大量）
/                \
```

| 层级 | 速度 | 粒度 | 维护成本 | 关注点 |
|------|------|------|---------|-------|
| 单元测试 | ⚡ 毫秒级 | 函数/类 | 低 | 代码逻辑正确性 |
| 集成测试 | 🚀 秒级 | 模块/服务 | 中 | 模块间协作 |
| E2E 测试 | 🐢 分钟级 | 全流程 | 高 | 用户场景 |

### 反模式：冰淇淋甜筒

```
         /\
        /  \       单元测试（极少 — 错误！）
       /    \
      /──────\
     /        \    集成测试（适中）
    /          \
   /────────────\
  /              \  E2E 测试（大量 — 错误！）
 /                \
```

> 过度依赖 E2E 测试会导致运行缓慢、维护成本高昂、反馈周期长。

## 按测试粒度分类

### 1. 单元测试

测试最小可测试单元（函数、方法、类），不依赖外部资源（数据库、网络、文件系统）。

**特征**：
- 隔离运行，依赖通过 Mock / Stub 替代
- 执行速度快，毫秒级反馈
- 发现问题后定位精确

```python
# 单元测试示例
def test_calculate_total():
    cart = ShoppingCart()
    cart.add_item(Item("apple", 10))
    cart.add_item(Item("banana", 20))
    assert cart.calculate_total() == 30
```

### 2. 集成测试

验证多个模块或外部系统之间的协作是否正确。

**特征**：
- 涉及真实数据库、网络 IO 或外部服务
- 比单元测试慢，但比 E2E 快
- 关注接口契约和数据流转

```python
# 集成测试示例
def test_user_repository():
    db = create_test_database()       # 真实数据库
    repo = UserRepository(db)
    repo.save(User("Alice", "alice@example.com"))
    user = repo.find_by_email("alice@example.com")
    assert user.name == "Alice"
```

### 3. 端到端测试 (E2E)

模拟真实用户操作，验证整个系统从 UI 到数据库的全链路。

**特征**：
- 最贴近用户视角
- 运行最慢，维护成本最高
- 通常覆盖核心业务流程

```python
# E2E 测试示例
def test_user_login_flow(page):
    page.goto("https://example.com/login")
    page.fill("#username", "alice")
    page.fill("#password", "pass123")
    page.click("#login-btn")
    assert page.is_visible("#dashboard")
```

## 按测试手段分类

### 黑盒测试

不关注内部实现，仅通过输入输出验证功能是否符合预期。

**适用**：集成测试、E2E 测试、API 测试

```
输入 → [ 被测系统 ] → 输出
         黑盒
```

**方法**：
- 等价类划分 — 将输入划分为有效/无效等价类，每类取一个代表值
- 边界值分析 — 重点关注边界条件（如 `min-1`, `min`, `max`, `max+1`）
- 决策表 — 适用于多个条件组合的场景
- 因果图 — 分析输入条件的组合与输出关系

```python
# 等价类划分示例
def test_discount():
    # 有效等价类：普通用户 (0-999), VIP (1000-4999), SVIP (5000+)
    assert calc_discount(500) == 0.0       # 普通
    assert calc_discount(2000) == 0.1      # VIP
    assert calc_discount(10000) == 0.2     # SVIP

def test_discount_boundary():
    # 边界值: 0, 999, 1000, 4999, 5000
    assert calc_discount(0) == 0.0         # 下边界
    assert calc_discount(999) == 0.0       # 上边界
    assert calc_discount(1000) == 0.1      # 下边界
```

### 白盒测试

基于内部代码结构设计测试用例，关注路径覆盖率。

**适用**：单元测试

**覆盖标准**（从弱到强）：

| 覆盖类型 | 说明 | 示例 |
|---------|------|------|
| 语句覆盖 | 每行代码至少执行一次 | 最弱，基本要求 |
| 分支覆盖 | 每个 if/else 分支至少执行一次 | 覆盖真假分支 |
| 条件覆盖 | 每个布尔子条件取真和假各一次 | 比分支覆盖更细 |
| 路径覆盖 | 所有可能路径至少执行一次 | 最强，组合爆炸 |

```python
def classify(score: int) -> str:
    if score >= 90:          # 分支1
        return "优秀"
    elif score >= 60:        # 分支2
        return "及格"
    else:                    # 分支3
        return "不及格"

# 分支覆盖测试
def test_classify():
    assert classify(95) == "优秀"    # 覆盖分支1
    assert classify(75) == "及格"    # 覆盖分支2
    assert classify(30) == "不及格"  # 覆盖分支3
```

### 灰盒测试

结合黑盒和白盒，既关注外部行为也关注部分内部逻辑，是最常见的工程实践。

## 按测试目的分类

### 功能测试

验证系统功能是否符合需求规格。

```
功能测试 = 输入条件 + 执行操作 + 验证输出
```

### 非功能测试

| 类型 | 关注点 | 工具 / 方法 |
|------|--------|------------|
| 性能测试 | 响应时间、吞吐量、资源使用 | JMeter, Locust, k6, wrk |
| 压力测试 | 系统在极限负载下的行为 | 逐步增加负载直到系统崩溃 |
| 稳定性测试 | 长时间运行是否正常 | 7×24 小时持续压测 |
| 安全测试 | 漏洞、权限、加密 | SAST / DAST, OWASP ZAP |
| 兼容性测试 | 浏览器、设备、操作系统 | Selenium Grid, BrowserStack |
| 可用性测试 | 用户体验、易用性 | 用户调研、A/B 测试 |

## 测试方法论

### TDD — 测试驱动开发

**红 → 绿 → 重构** 循环：

1. 编写一个**失败**的测试（红）
2. 编写刚好能让测试通过的代码（绿）
3. 重构代码，保持测试通过

```python
# 第1步：写一个失败的测试
def test_is_palindrome():
    assert is_palindrome("racecar") == True

# 第2步：写最简实现
def is_palindrome(s: str) -> bool:
    return s == s[::-1]

# 第3步：重构（这里不需要）
```

### BDD — 行为驱动开发

基于自然语言描述的测试，促进团队协作：

```gherkin
Feature: 用户登录
  Scenario: 使用正确密码登录
    Given 用户已注册
    And 用户名为 "alice"，密码为 "pass123"
    When 用户在登录页面输入用户名和密码
    And 点击登录按钮
    Then 用户应跳转到首页
    And 页面显示 "欢迎回来，alice"
```

Python 实现（pytest-bdd）：

```python
# test_login.py
from pytest_bdd import scenarios, given, when, then

scenarios("login.feature")

@given("用户已注册")
def registered_user():
    create_user("alice", "pass123")

@when("用户在登录页面输入用户名和密码")
def enter_credentials(login_page):
    login_page.fill_username("alice")
    login_page.fill_password("pass123")

@when("点击登录按钮")
def click_login(login_page):
    login_page.click_login()

@then("用户应跳转到首页")
def verify_url():
    assert current_url() == "https://example.com/dashboard"
```

### Fuzzing — 模糊测试

通过随机或变异生成输入，发现边界情况和安全漏洞：

```go
func FuzzParsePhone(f *testing.F) {
    f.Add("13800138000")
    f.Add("abc")
    f.Fuzz(func(t *testing.T, input string) {
        result, err := ParsePhone(input)
        if err == nil {
            // 解析成功，验证结果格式
            if len(result) != 11 {
                t.Errorf("expected 11 digits, got %d", len(result))
            }
        }
    })
}
```

## 常见误区

| 误区 | 正确做法 |
|------|---------|
| 追求 100% 覆盖率 | 覆盖率是手段不是目标，重点测核心逻辑和风险区域 |
| 测试依赖实现细节 | 测试行为而非实现，重构时测试不应频繁失效 |
| 不写测试直接进入 E2E | 遵循测试金字塔，以单元测试为主 |
| 测试之间互相依赖 | 每个测试应独立、可重复、可并行执行 |
| 测试代码不维护 | 测试代码和生产代码同等对待，保持整洁 |
| 只测 happy path | 必须覆盖异常路径和边界条件 |

## Tips

| 提示 | 说明 |
|------|------|
| FIRST 原则 | Fast / Independent / Repeatable / Self-checking / Timely |
| 每写一个测试前先看它失败 | 避免误判测试永远通过（假阳性） |
| 一个测试只测一个关注点 | 断言失败时能立即定位问题 |
| 使用有意义的测试名称 | `test_withdraw_reduces_balance` 而非 `test_1` |
| 避免在测试中使用 sleep | 改为轮询或等待异步回调 |
| 使用测试替身隔离外部依赖 | Mock / Stub / Fake / Spy / Dummy |
